# 06.06 화

## 🚶‍♂️ 오늘 한 것

- 포스팅
  - `REST API란 무엇이고, 반드시 지켜야만 할까?`

## 📚 배운 것

- REST API
  - REST API
    - REST 아키텍처를 따르는 API
    - URI를 통해 사용할 자원을 명세하고, HTTP Method로 행위를 표현하는 스타일
    - 요청형식만으로 어떤 요청인지 추론이 가능 👉 다수의 개발자와 협업 가능
  - REST
    - 웹 리소스를 어떻게 지정하고, 접근할 것인지를 제약하는 소프트웨어 아키텍처
    - 아래 원칙들로 인해 효율적이고 확장적인 서비스 개발 가능 
    - 원칙
      - Uniform Interface
        - 서버가 일관된 형식으로 정보를 응답하는 것 
      - Stateless
      - Client-Server Separation
      - Cacheable
      - ...
  - RESTful API
    - REST 제약, 특히 HATEOS(하이퍼링크를 통해 리소스의 상태 전이) 등을 만족시키기 어려우므로 최대한 REST를 따르는 RESTful API를 많이 사용
    - RESTful API를 이용하여 구축한 웹을 `RESTful 웹 서비스`라고 한다. 
- 캐시
  - 자주 쓰이는 문서의 사본을 저장하는 HTTP 장치
  - 웹 요청이 캐시에 도착했을 때, 캐시된 로컬 사본이 존재하면 요청을 서버로 보내지 않고 요청에 대한 응답을 캐시에서 제공 👉 서버 성능 향상

## 📚 배운 것

### 테스트 코드 

- **요구사항이 주어졌을 때, 클라이언트 개발자 또는 기획자분께 암묵적이거나 아직 드러나지 않은 요구사항이 있는지 확인**해야한다. 
- **해피 케이스, 예외 케이스** 
  - 요구사항이 주어졌을 대, 해피 케이스와 예외 케이스로 구분해서 생각
  - 예외 케이스의 경우, `경계값 테스트`를 하자
    - 예) 잔액이 음수로 입력된다면?
    - 예) 수량이 1개 미만으로 입력된다면?
- **테스트 하기 쉬운 영역과 어려운 영역을 구분하자**
  - `어려운 영역`
    - 관측할 때 마다 다른 값에 의존하는 경우
      - 랜덤값, 전역 변수/함수, 사용자 입력 등
    - 외부 세계에 영향을 주는 코드 
      - 표준 출력, 메시지 발송, 데이터베이스에 기록하기 등
  - `쉬운 영역`
    - 순수 함수
      - 같은 입력에 대해 항상 같은 결과
      - 외부 세상과 단절된 형태

### Network

- 쿠키와 세션
  - HTTP가 갖는 Connectionless, Stateless 특성 때문에 클라이언트의 정보를 유지할 수 없는 단점을 해결하기 위해 사용하는 웹 기술
  - 차이: 서버 자원을 활용하느냐 안 하느냐
  - `쿠키`
    - **클라이언트의 브라우저에 `Key-Value` 형태로 저장되는 로컬 파일**
    - 유효시간 내 브라우저가 종료되어도 계속 유지
    - 아이디/비밀번호 값 저장 등에 사용
    - 보안에 약함
    - 동작과정
      - 클라이언트 요청에 대해 서버는 Response Header에 쿠키를 담아서 응답
      - 클라이언트의 브라우저의 `쿠키 스토리지`에 쿠키를 저장
      - 다음 요청마다 Request Header에 쿠키를 담아 응답
  - `세션`
    - 기본적인 구현은 쿠키와 동일 
    - 서버에서 세션 ID를 발급해서, 쿠키 스토리지에 저장.
    - 세션 같은 경우 브라우저 종료 시에나, 서버에서 자체적으로 세션 만료 기간을 설정 가능하기 때문에, 쿠키가 갖는 보안 위험을 해결
- Web Server, WAS, 웹 컨테이너, 서블릿, 멀티 쓰레드
  - Web Server 
    - 요청에 대해 정적인 컨텐츠(HTML, CSS, JS)를 응답하는 서버. Apache
    - WAS의 부하를 줄여주기 위해서, WAS 로 요청이 가기 이전에 설치하여 사용.
  - WAS 
    - 요청에 대해 동적인 컨텐츠(비즈니스 로직 수행 후)를 응답. Tomcat 
  - 웹 컨테이너 
    - WAS 내부에서 서블릿을 관리
    - 서블릿 호출
    - 요청에 대해 쓰레드 할당 등의 작업을 한다.
  - 서블릿
    - 개발자가 비즈니스 로직에만 집중할 수 있게한다.
    - HTTP Request 메시지는 결국 단순 텍스트인데 이걸 개발자들이 매번 파싱하면 비효율적
    - HTTP Request 메시지 파싱, Response 메시지 생성 등의 작업 
  - 멀티 스레드
    - 서블릿은 스레드가 실행을 시킨다. 
      - main 스레드가 자바 코드를 실행시키는 것처럼
    - `스레드 풀`
      - 여러 클라이언트의 요청을 처리하기 위해 WAS는 멀티 스레드로 동작
      - 멀티스레드를 관리하는 것이 스레드 풀
      - 스레드를 여러 개 생성해두고 요청이 있을 때마다 할당. 처리가 끝나면 반납.
- Redirect Vs. Forward
  - 둘 다 새로운 웹 페이지로 이동하기 위한 기술
  - `Forward`
    - 같은 웹 컨테이너에서의 이동.
    - 클라이언트는 이 이동을 알아차릴 수 없음. 
      - URL은 최초 호출한 URL만 표기 
    - HTTP Request, Response 유지 
    - 게시글 생성 완료 후에 `새로고침`을 하게 되면, 이전 요청의 정보가 남아있으므로 같은 게시글이 중복되서 생성 됨. 👉 단순 목록 조회 등에 사용 
  - `Redirect`
    - 다른 웹 컨테이너에서의 이동
    - 처음 요청에 대해 서버가 URL을 지정해주면, 클라이언트 브라우저는 해당 URL로 다시 요청
    - HTTP Request, Response 정보 유지 X
    - 게시글 생성 완료 후 `새로고침`을 해도 게시글 중복 생성되지 않음
- HTTP Referer Header
  - HTTP 헤더에 있는 Referer라는 값.
  - URL 이동 시 이전에 있던 URL이 Referer에 기록이 된다. 
  - 어디에서 유입되었는지 방문 경로를 확인할 수 있다.
